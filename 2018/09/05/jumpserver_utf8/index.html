Jumpserver命令记录入库失败处理

发表于 2018-09-05 | 分类于 运维

背景

某一天，zabbix突然报主机的磁盘空间告警，检查发现jumpserver服务后题疯狂打日志，半天日志大概能涨几个G空间，日志报错内容如下：

在jumpserver项目的github上我提过一个issues：https://github.com/jumpserver/jumpserver/issues/1773

处理过程

1、查看日志报错分析为有记录插入mysql数据库失败，觉得可能是mysql字符集的问题，然后就把mysql字符集修改为了全部utf8，并重启了mysql数据库。然后肯定就失败了，这么简单怎么可能有这篇博客。

2、就把上面插库失败的字符用ipython解码得到原来是表情字符：

3、mysql原始的utf8默认是3个字节的，但表情字符是使用4个字节的，所以utf8并不能存表情字符，要换成utf8mb4，然后就开始换了：

4、然后还是没解决，后台日志还是显示不能正常插入。我以为可能是对于原先创建的表修改字符集生效有问题，然后就经过了一段时间的删库重建数据库的过程，结局也很感人，依然没成功。

5、突然我想到我手工插下库试试，居然TM的成功了，还能查出来：

6、居然手工能插入，那应该是jumpserver程序插库时有问题，我就调试看了下jumpserver插库部分代码，发现django/db/backends/mysql/base.py中的DatabaseWrapper类中有个kwargs的属性，里面有个key叫charset，默认值是utf8，然后我就在jumpserver的settings.py中修改数据库配置，添加了'OPTIONS': {'charset': 'utf8mb4'}：

7、奇迹出现了，后台日志不报错了，检查命令记录成功入库了：

总结解决方法

1、修改Mysql的字符集编码为utf8mb4

2、修改jumpserver的数据库设置选贤，添加字符编码格式为utf8mb4

额外：修改Mysql字符集的一些命令

utf8mb4和utf8

utf8mb4最明显的好处是解决了苹果挖的坑-推广了emoji表情，utf8mb4解决了MySQL数据库存储emoji表情的问题。为了后续数据正常读取写入，尽量使用utf8mb4，虽然字符类型字段占用空间会比utf8多，但能正确保存数据。

utf8mb4是utf8的超集，理论上由utf8升级到utf8mb4字符编码没有任何兼容问题。

utf8mb4是MySQL5.5.3版本之后支持的字符集，所以保证mysql版本>=5.5.3

查看字符集

1、查看当前系统默认的字符集
SHOW VARIABLES WHERE Variable_name LIKE 'character\_set\_%' OR Variable_name LIKE 'collation%';

2、查看database的字符集

show create database database_name

3、查看表格的所有字段的字符集

show full columns from table_name;

修改字符集

1、修改当前系统的默认的字符集

网上教程都是修改/etc/my.cnf中，但不同系统mysql的配置文件可能不同，比如我的系统是ubuntu 16，然后修改配置文件就要修改三个：

自己可以试着找一下，就是修改修改上面三个部分字符集编码为utf8mb4，修改完后需要重启数据库，然后进入数据库查看一下修改完成没。

2、修改database的字符集

ALTER DATABASE database_name CHARACTER SET = utf8mb4 COLLATE = utf8mb4_unicode_ci;

3、修改table的字符集

  • 只修改表默认的字符集ALTER TABLE table_name DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

  • 修改表默认的字符集和所有字符列的字符集ALTER TABLE table_name CONVERT TO CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
